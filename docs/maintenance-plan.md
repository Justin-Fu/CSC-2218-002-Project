3. Versioning Strategy
    While I have not started on finalizing a versioning strategy, I believe that it is the best case to keep the versioning strategy simple and reliable, which will help keep track of project versions. I plan to have a version format of: v1.0.0, with each number representing a different amount of change done to the project. The first number will be used for major changes to the project. The second, or middle, number will be used for most average and typical changes in the project, which does not need to change the first number. The third, or last, number will be used for minor changes and small fixes that will not make any larger changes. This numbering system will help keep track of the versioning history, rather, than relying on only using 1 number (example: v1, v2, v3) as a versioning history, or using the current day (2026-04-15, 2026-04-16, 2026-04-17) as a versioning history, as these two versioning methods can be prone to more errors and confusion.
